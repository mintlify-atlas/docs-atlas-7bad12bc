---
title: "Get started with ShedLock"
description: "Add distributed locking to your Spring Boot scheduled tasks in four steps using ShedLock with a JDBC database or Redis as the lock provider backend."
---

ShedLock prevents your Spring scheduled tasks from running concurrently across multiple application nodes. This guide walks you through adding it to a Spring Boot application backed by a relational database. The entire setup takes four steps: add the dependency, enable locking in your configuration, create a `LockProvider` bean, and annotate the tasks you want to protect.

<Steps>
  <Step title="Add the Maven dependency">
    Add `shedlock-spring` to your `pom.xml`. This is the only dependency required for the Spring integration itself; you will also add a lock provider dependency in step 3.

    ```xml
    <dependency>
        <groupId>net.javacrumbs.shedlock</groupId>
        <artifactId>shedlock-spring</artifactId>
        <version>7.7.0</version>
    </dependency>
    ```
  </Step>

  <Step title="Enable scheduler lock">
    Add `@EnableSchedulerLock` to your Spring configuration class alongside the existing `@EnableScheduling` annotation. The `defaultLockAtMostFor` value sets how long a lock is held if the node holding it crashes before releasing it.

    ```java
    import net.javacrumbs.shedlock.spring.annotation.EnableSchedulerLock;
    import org.springframework.context.annotation.Configuration;
    import org.springframework.scheduling.annotation.EnableScheduling;

    @Configuration
    @EnableScheduling
    @EnableSchedulerLock(defaultLockAtMostFor = "10m")
    public class SchedulerConfiguration {
        // other beans
    }
    ```
  </Step>

  <Step title="Configure a LockProvider bean">
    ShedLock needs an external store to coordinate locks across nodes. Add the provider dependency and declare a `LockProvider` bean.

    <Tabs>
      <Tab title="JDBC (recommended)">
        First, create the `shedlock` table in your database. The `name` column must be the primary key.

        ```sql
        -- MySQL / MariaDB
        CREATE TABLE shedlock(
            name        VARCHAR(64)   NOT NULL,
            lock_until  TIMESTAMP(3)  NOT NULL,
            locked_at   TIMESTAMP(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
            locked_by   VARCHAR(255)  NOT NULL,
            PRIMARY KEY (name)
        );

        -- Postgres
        CREATE TABLE shedlock(
            name        VARCHAR(64)  NOT NULL,
            lock_until  TIMESTAMP    NOT NULL,
            locked_at   TIMESTAMP    NOT NULL,
            locked_by   VARCHAR(255) NOT NULL,
            PRIMARY KEY (name)
        );
        ```

        Add the JDBC provider dependency:

        ```xml
        <dependency>
            <groupId>net.javacrumbs.shedlock</groupId>
            <artifactId>shedlock-provider-jdbc-template</artifactId>
            <version>7.7.0</version>
        </dependency>
        ```

        Declare the `LockProvider` bean:

        ```java
        import net.javacrumbs.shedlock.provider.jdbctemplate.JdbcTemplateLockProvider;
        import org.springframework.jdbc.core.JdbcTemplate;
        import javax.sql.DataSource;

        @Bean
        public LockProvider lockProvider(DataSource dataSource) {
            return new JdbcTemplateLockProvider(
                JdbcTemplateLockProvider.Configuration.builder()
                    .withJdbcTemplate(new JdbcTemplate(dataSource))
                    .usingDbTime()
                    .build()
            );
        }
        ```

        <Tip>
          Always use `.usingDbTime()`. It makes ShedLock use the database server's clock instead of the application server's clock, which eliminates locking failures caused by clock drift between nodes. It also uses database-engine-specific SQL that prevents INSERT conflicts.
        </Tip>
      </Tab>
      <Tab title="Redis">
        Add the Redis provider dependency:

        ```xml
        <dependency>
            <groupId>net.javacrumbs.shedlock</groupId>
            <artifactId>shedlock-provider-redis-spring</artifactId>
            <version>7.7.0</version>
        </dependency>
        ```

        Declare the `LockProvider` bean:

        ```java
        import net.javacrumbs.shedlock.provider.redis.spring.RedisLockProvider;
        import org.springframework.data.redis.connection.RedisConnectionFactory;

        @Bean
        public LockProvider lockProvider(RedisConnectionFactory connectionFactory) {
            return new RedisLockProvider(connectionFactory, "production");
        }
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Annotate your scheduled tasks">
    Add `@SchedulerLock` to each method you want to protect. Only methods annotated with `@SchedulerLock` are locked — ShedLock ignores all other `@Scheduled` methods.

    ```java
    import net.javacrumbs.shedlock.core.LockAssert;
    import net.javacrumbs.shedlock.spring.annotation.SchedulerLock;
    import org.springframework.scheduling.annotation.Scheduled;
    import org.springframework.stereotype.Component;

    @Component
    public class ReportingTask {

        @Scheduled(cron = "0 */15 * * * *")
        @SchedulerLock(name = "reportingTask", lockAtMostFor = "14m", lockAtLeastFor = "14m")
        public void generateReport() {
            // Assert that the lock is held — catches AOP misconfiguration early
            LockAssert.assertLocked();

            // task logic here
        }
    }
    ```

    <Warning>
      Set `lockAtMostFor` to a value significantly longer than the maximum expected execution time. If a task runs longer than `lockAtMostFor`, the lock expires and another node may start the same task concurrently, producing unpredictable results. The default value from `@EnableSchedulerLock` is used when `lockAtMostFor` is not specified on the annotation.
    </Warning>
  </Step>
</Steps>
